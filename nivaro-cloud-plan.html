🔑
DB_CLIENT is the only dial you turn — no code changes needed
Nivaro's dialect-agnostic design means you set DB_CLIENT=mssql|pg|mysql2 in the environment and point at your cloud's managed database. There is no migration, no code rewrite. Azure works best with mssql and Elastic Pools. AWS and GCP work best with pg and their Postgres-flavoured managed databases. Indie stacks work with any of the three.

AWS and GCP are equally viable with DB_CLIENT=pg pointing at Aurora Serverless v2 or Cloud SQL — both have shared-compute multi-tenant models comparable to Elastic Pools. The Simple/Indie path works for any dialect and keeps infra costs under $200/mo for your first 50 customers.

03 — Database Strategy
Section 03
Database Strategy
Nivaro creates real SQL tables for user-defined collections — two customers both creating "articles" would collide in a shared database. DB-per-tenant eliminates this without changing the schema builder. The dialect (mssql / pg / mysql2) is a deployment-time decision driven purely by which managed database your cloud provider offers cheaply.
💡
Why DB-per-tenant is the right architecture for Nivaro
When a customer creates a collection called "articles", Nivaro runs CREATE TABLE articles. In a shared database, two customers with the same collection name collide. Separate databases mean each customer's "articles" is just "articles" — no prefixing, no schema builder changes, no cross-tenant data risk. The dialect is set via DB_CLIENT — the rest of Nivaro doesn't care which one.

Connection Routing (all providers — same pattern)

// Runs before every request — resolves the correct tenant DB + dialect async function resolveTenantDb(req, reply) { const subdomain = req.hostname.split('.')[0] // 'acme' from acme.nivaro.io const cacheKey = `tenant:${subdomain}` let cached = await redis.get(cacheKey) if (!cached) { const tenant = await metaDb('cloud_tenants').where({ subdomain }).first() if (!tenant) return reply.code(404).send({ error: 'Not found' }) cached = JSON.stringify({ client: tenant.db_client, conn: tenant.db_connection_string }) await redis.setex(cacheKey, 300, cached) // 5-min cache → ~1ms lookup } const { client, conn } = JSON.parse(cached) req.tenantDb = getOrCreatePool(client, conn) // Map keyed by connStr — never creates per-request } // Pool factory — reuses existing Knex instance for same connection string const pools = new Map() function getOrCreatePool(client, conn) { if (!pools.has(conn)) pools.set(conn, knex({ client, connection: conn, pool: { min: 0, max: 5 } })) return pools.get(conn) }

05 — Tenant Provisioning
Section 05
Tenant Provisioning Flow
Signup to live tenant in under 60 seconds. Orchestrated by Inngest — fully automated, no manual steps. The provider-specific step is DB creation (Step 3); everything else is identical.
1
Customer pays via Stripe Checkout
Stripe fires checkout.session.completed webhook. Handler validates signature, immediately returns 200, then fires an Inngest event.
2
Inngest tenant/provision function starts
Durable, retryable, observable. Each step below is an Inngest step — failures retry automatically without re-running completed steps.
3
Database created — provider-specific
Call the cloud provider's management API to create the tenant database:
Azure
azure.sql.databases
.beginCreateOrUpdate(
  resourceGroup, server,
  `tenant_${slug}`,
  { elasticPoolId: pool.free })
AWS (DB_CLIENT=pg)
rds.createDBCluster({
  DBClusterIdentifier:
    `tenant-${slug}`,
  Engine: 'aurora-postgresql',
  EngineMode: 'provisioned',
  ServerlessV2ScalingConfig:
    { MinCapacity: 0.5, MaxCapacity: 4 } })
GCP (DB_CLIENT=pg)
// Multiple DBs on one Cloud SQL instance
sql.databases.insert({
  project, instance,
  requestBody: {
    name: `tenant_${slug}` }})
Indie (DB_CLIENT=pg)
// Railway / self-hosted Postgres
await db.raw(
  `CREATE DATABASE
   tenant_${slug}`)
4
Migrations run on fresh DB
Connect to new DB with provisioning-scoped Knex instance, run migrate.latest(). All Nivaro system tables created. ~10–20 seconds.
5
Admin user & workspace seeded
Insert nivaro_workspaces, nivaro_roles (admin), and nivaro_users with the customer's email and a temporary static token for first login.
6
Tenant registered in Meta DB + DNS provisioned
Insert into cloud_tenants with the DB connection string — subdomain is now live. Cloudflare API creates the CNAME for {slug}.nivaro.io.
7
Welcome email sent
Nivaro's own mail service sends login URL + temp credentials. Total: ~45–60 seconds.

06 — Migration Strategy
Section 06
Migration Strategy
With N tenant databases, every schema migration must fan out to every tenant. The solution is an Inngest function that runs on deploy and nightly — applying pending migrations to all active tenants in sequence.
⚠️
This is the #1 operational challenge of DB-per-tenant
A single-DB deploy takes seconds. With 200 tenant DBs, a migration runner iterates every DB — fine for additive changes, but destructive changes (DROP COLUMN, rename, type change) need the expand/contract pattern across two separate deploys.

The Migration Runner (Inngest)

// Fires on every deploy + nightly at 02:00 UTC as catchup inngest.createFunction( { id: 'run-tenant-migrations', retries: 3 }, [{ event: 'deploy/completed' }, { cron: '0 2 * * *' }], async ({ step }) => { const tenants = await step.run('fetch-tenants', () => metaDb('cloud_tenants').where({ status: 'active' }) ) for (const tenant of tenants) { await step.run(`migrate-${tenant.slug}`, async () => { const db = knex({ client: 'mssql', connection: tenant.db_connection_string }) try { await db.migrate.latest() await metaDb('cloud_tenant_migrations') .insert({ tenant_id: tenant.id, status: 'ok', applied_at: new Date() }) } finally { await db.destroy() } }) } } )

Expand / Contract for Destructive Changes

Deploy 1 (Expand)Gap — code reads bothDeploy 2 (Contract)
Add new_column (nullable)App writes to both columnsDrop old_column
Create new table + copy dataDual-write to both tablesDrop old table
Add new indexBoth indexes existDrop old index

✅ Safe for immediate fan-out

  • Add new nullable column
  • Add new index (ONLINE=ON)
  • Create new table
  • Add NOT NULL with default
  • Extend enum / add new option

⛔ Use expand/contract instead

  • DROP COLUMN or DROP TABLE
  • Rename column or table
  • Change column data type
  • Add NOT NULL to existing rows
  • Remove an enum option
✅
Build a migration health dashboard
The meta DB cloud_tenant_migrations table tracks status per tenant. A simple internal admin page showing green/amber/red per tenant makes it immediately visible if any tenant is stuck on an old migration version. Failed migrations auto-retry via Inngest's built-in retry logic.

10 — Operations & Security
Section 10
Operational Playbook & Security
What needs to be running before you take your first paid customer — provider-agnostic.

Monitoring Stack

📊 Required before launch

  • Better Stack — status page + uptime per subdomain
  • Sentry — JS + Node error tracking (free tier)
  • PagerDuty — on-call alerts for P0/P1
  • Migration status dashboard (internal)
  • Provider-native metrics (App Insights / CloudWatch / Cloud Monitoring)

Backup Strategy

💾 Data Protection

  • Azure: SQL auto-backups, 7-day PITR included
  • AWS: RDS automated snapshots (7–35 days)
  • GCP: Cloud SQL automated backups daily
  • Indie: pg_dump / sqlcmd cron → offsite storage
  • Blob/S3/GCS: geo-redundant storage for files
  • Meta DB daily backup to separate account
  • Quarterly restore drill — test before needed

Incident Response

🚨 SLA levels

  • P0 full outage: 15min response + status page update
  • P1 degraded: 1hr response + affected tenants notified
  • P2 single tenant: 4hr Business, 24hr others
  • Public post-mortems for P0/P1 — builds trust

Security Baseline

🔐 Before first customer

  • DB connections over TLS (DB_ENCRYPT=true — already set)
  • Connection strings in secrets manager (Key Vault / Secrets Manager / Secret Manager)
  • Tenant DB user has minimal permissions — no DROP TABLE
  • Separate provisioning service account (CREATE DATABASE only)
  • Stripe webhook signature validation
  • Rate limiting on all public routes
  • CORS locked to *.nivaro.io

Compliance Checklist

📋 Legal requirements

  • Privacy Policy — required before launch
  • DPA (Data Processing Agreement) — EU customers
  • Terms of Service with acceptable use policy
  • GDPR data export + deletion (already built in Nivaro)
  • SOC 2 Type II — start Month 4–5

Tenant Isolation Guarantee

🏠 DB-per-tenant means

  • Bug in tenant A cannot read tenant B data
  • Noisy tenant cannot degrade others (pool throttling)
  • Compromised credentials scoped to one tenant
  • Delete tenant = drop database, no data leaks
  • Satisfies most enterprise security questionnaires
🚀
You're further along than you think
Nivaro already ships: multi-workspace isolation, usage counters, RBAC, audit logs, revision history, notifications, OIDC auth, retention/redaction, PDF templates, and rate limiting. The gap to a launchable SaaS is provisioning automation, file storage swap, tenant routing middleware, and Stripe integration — realistically 6–8 weeks of focused engineering.
